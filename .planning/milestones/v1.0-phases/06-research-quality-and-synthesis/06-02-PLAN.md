Step 14 must implement the following:

**14a: Compute Gate Criteria from Evaluator Output**
- Count from state.yml question statuses (set in Step 12):
  - `total_questions`: count of all questions
  - `covered_count`: count where status = "covered"
  - `partial_count`: count where status = "partial"
  - `not_covered_count`: count where status = "not_covered"
  - `unavailable_count`: count where status = "unavailable_source"
  - `p0_questions`: subset with priority = "P0"
  - `p0_not_covered`: P0 questions where status = "not_covered"
  - `unresolved_unavailable`: UNAVAILABLE-SOURCE questions where user has NOT yet decided (should be 0 after Step 12c)

**14b: Evaluate MUST Criteria**
Following the G1 gate pattern from scope SKILL.md Step 9:

```
MUST criteria (all must pass for Go):
M1: Every question has been assessed (covered + partial + not_covered + unavailable = total)
M2: Majority of questions rated COVERED (covered_count > total / 2)
M3: All P0 questions rated COVERED or PARTIAL (p0_not_covered == 0)
M4: No UNAVAILABLE-SOURCE questions remain unresolved (unresolved_unavailable == 0)
```

For each MUST criterion, record pass/fail with evidence (e.g., "M2: PASS — 7/10 covered (70% > 50%)").

**14c: Evaluate SHOULD Criteria**

```
SHOULD criteria (failures produce advisory, not block):
S1: All questions rated COVERED (covered_count == total)
S2: No PARTIAL ratings remain (partial_count == 0)
S3: All evidence requirements MET (no PARTIALLY MET in evaluator output)
```

For each SHOULD criterion, record pass/fail with evidence.

**14d: Determine Gate Outcome**
- **Go**: All MUST pass AND all SHOULD pass
- **Go-with-advisory**: All MUST pass, one or more SHOULD fail
- **Recycle**: Any MUST fails (except M4 which was handled in Step 12c)
- **Override**: Only available when user explicitly requests it (not auto-determined)

**14e: Display Gate Result**
Show gate evaluation results in the same format as G1:
```
## G2 Gate Evaluation

MUST Criteria:
  M1: {PASS|FAIL} — {evidence}
  M2: {PASS|FAIL} — {evidence}
  M3: {PASS|FAIL} — {evidence}
  M4: {PASS|FAIL} — {evidence}

SHOULD Criteria:
  S1: {PASS|FAIL} — {evidence}
  S2: {PASS|FAIL} — {evidence}
  S3: {PASS|FAIL} — {evidence}

Outcome: {Go | Go-with-advisory | Recycle}
```

Proceed to Step 15 for outcome handling.

**Step 15: Gate Outcome Handling**

**15a: Record Gate History**
- Follow backup-before-write pattern
- Append to gate_history in state.yml:
  ```yaml
  - gate: "G2"
    timestamp: "{ISO 8601 UTC}"
    outcome: "{go|go_advisory|recycle|override}"
    must_passed: {count}
    must_failed: {count}
    should_passed: {count}
    should_failed: {count}
    notes: "{brief summary}"
    overridden: false
  ```

**15b: Recycle Escalation Logic (GATE-04)**
- Count previous G2 recycle outcomes in gate_history
- Escalation messaging:
  - **1st recycle (recycle_count == 0):** Informational — "Some questions need additional evidence. Here's what's missing:" + show gap details from evaluator. Then ask user: "Approve gap-fill research? (yes / adjust questions / override)"
  - **2nd recycle (recycle_count == 1):** Suggest adjustment — "This is the second recycle. Consider adjusting expectations:" + show which questions are persistently PARTIAL/NOT COVERED + suggest: lower priority of stubborn questions, accept as advisory, or try different sources. Ask: "Approve gap-fill / Adjust / Override?"
  - **3rd+ recycle (recycle_count >= 2):** Recommend override — "This is recycle #{N+1}. Recommend overriding the gate:" + explain that remaining gaps may not be resolvable through additional research + suggest: override with documented gaps flowing to design as advisory. Ask: "Override (recommended) / One more gap-fill attempt?"

**15c: Handle Go Outcome**
- Display: "G2 gate passed. Research is sufficient for design."
- Proceed to Step 17 (synthesis generation)

**15d: Handle Go-with-Advisory Outcome**
- Pause and show the user which SHOULD criteria failed and which questions have weak areas (PARTIAL ratings, PARTIALLY MET evidence requirements)
- Present via freeform prompt: "Some areas are weak but not blocking. You can: (1) Proceed with advisory — weak areas documented in SYNTHESIS.md (2) Run gap-fill to strengthen these areas before synthesis"
- If user chooses to proceed: record advisory data for the synthesizer (list of SHOULD failures and PARTIAL question details). Proceed to Step 17.
- If user chooses gap-fill: treat as Recycle — proceed to Step 16

**15e: Handle Recycle Outcome**
- Apply escalation messaging from 15b based on recycle count
- Show gap details: for each PARTIAL/NOT COVERED question, display question text, current status, and gap_details (what evidence is still missing)
- Wait for user decision:
  - **Approve gap-fill:** Proceed to Step 16
  - **Adjust:** User can reprioritize questions (lower priority) or mark questions as "accept gap." Update state.yml accordingly, then re-run Step 14 with updated statuses
  - **Override:** Proceed to 15f

**15f: Handle Override Outcome**
- Record in gate_history with `overridden: true` and severity level
- Write override context to `.expedite/research/override-context.md`:
  ```
  # G2 Override Context
  Timestamp: {ISO 8601 UTC}
  Severity: {low|medium|high based on how many MUST criteria failed}
  Recycle count: {N}

  ## Overridden Gaps
  {For each NOT COVERED/PARTIAL question:}
  - {question_id}: {question_text}
    Status: {status}
    Missing: {gap_details}
    Impact: {which DA is affected}

  ## Design Phase Advisory
  The following decision areas have insufficient evidence. Design decisions for these areas should be flagged as lower confidence.
  {List affected DAs}
  ```
- Transition phase to `"research_in_progress"` (synthesis will transition to `"research_complete"`)
- Proceed to Step 17

**Step 16: Gap-Fill Dispatch**

**16a: Filter to Deficient Questions**
- From state.yml, collect questions where status is "partial" or "not_covered"
- Include any newly approved discovered questions (from Step 13) that have status "pending"
- Exclude UNAVAILABLE-SOURCE questions (user already decided in Step 12c)

**16b: Re-Batch by Decision Area**
- Group deficient questions by `decision_area` (user decision — different from first-round source-affinity batching)
- Each DA becomes one batch
- Within each batch, include: question_id, question_text, evidence_requirements, gap_details (from evaluator — tells agents what's still missing), existing evidence file paths (so agents can read what was already found)

**16c: Increment Research Round**
- Update state.yml `research_round` to N+1 (backup-before-write)
- Create directory `.expedite/research/round-{N+1}/` for supplement files
- Transition phase to `"research_in_progress"` (from `"research_recycled"`)

**16d: Dispatch Gap-Fill Agents**
- Use the SAME dispatch pipeline as Phase 5 (Steps 7-11), but with:
  - Narrowed question set (only deficient questions)
  - DA-based batches instead of source-affinity batches
  - Additional `<gap_context>` block injected into template input_data section containing the evaluator's gap_details for each question
  - Output path: `.expedite/research/round-{N}/supplement-{DA-slug}.md` (additive — never overwrite originals)
- Parallel dispatch with same 3-agent max concurrency
- Template assembly: use the same per-source templates, fill same placeholders, add gap_context injection
- Gap-fill agents produce ONLY supplement evidence targeting the specific gaps — not duplicate existing findings

**16e: After Gap-Fill Completes**
- Dynamic question discovery happens in gap-fill rounds too (user decision) — collect any new proposed questions and append to proposed-questions.yml
- Update evidence_files in state.yml for each question with new supplement file paths
- Display gap-fill summary: "{N} supplement files produced across {M} DAs"
- Return to Step 12 to re-assess sufficiency with the new evidence

Note: The loop is Step 12 -> Step 13 -> Step 14 -> Step 15 -> Step 16 -> back to Step 12. This continues until G2 passes or user overrides.
