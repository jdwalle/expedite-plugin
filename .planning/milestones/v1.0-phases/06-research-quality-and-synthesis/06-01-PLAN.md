Step 12 must implement the following, using the same linear-step prose format established in Steps 1-11:

**12a: Input Assembly for Evaluator**
- Read `.expedite/scope/SCOPE.md` to extract: each question's evidence requirements, each DA's depth calibration (Deep/Standard/Light), readiness criteria
- Read each evidence file listed in state.yml `evidence_files` arrays for each question
- If gap-fill rounds exist, also read `round-{N}/supplement-*.md` files from `.expedite/research/`
- Assemble the `{{questions_with_evidence}}` placeholder block for the evaluator: for each question, include question_id, question_text, decision_area, da_depth, evidence_requirements, and the FULL evidence content from relevant evidence files
- DO NOT include dispatch metadata, agent reasoning, Phase 5 preliminary statuses, or batch configuration in the evaluator input — structural separation is a locked user decision (GATE-07 anti-bias)
- Only include evidence files and scope artifacts

**12b: Invoke Sufficiency Evaluator (Inline Reference)**
- Read the evaluator template from `skills/research/references/prompt-sufficiency-evaluator.md` (use Glob with `**/prompt-sufficiency-evaluator.md` if the direct path fails)
- This is an inline reference (no frontmatter, no Task() dispatch) — the orchestrator fills placeholders and applies the evaluation logic directly in the main session
- Fill placeholders: `{{project_name}}` from state.yml, `{{intent}}` from state.yml, `{{questions_with_evidence}}` from the assembled block
- Apply the `<if_intent_product>` or `<if_intent_engineering>` conditional blocks based on intent
- Execute the evaluation following the template's instructions exactly: assess each question individually across 3 dimensions (Coverage, Corroboration, Actionability), apply DA depth calibration for corroboration thresholds, assign categorical rating per the template's rules
- The evaluator's quality gate (anti-bias self-check) must be performed before finalizing

**12c: UNAVAILABLE-SOURCE Short-Circuit**
- For any question rated UNAVAILABLE-SOURCE by the evaluator: surface immediately to the user with AskUserQuestion
- Present options: "Accept gap (advisory in synthesis)" / "Suggest alternative source" / "Override and proceed"
- If user suggests alternative source: update the question's source_hints in state.yml, do NOT re-run sufficiency — the question will be handled in gap-fill if G2 recycles
- If user accepts gap: record the decision, question keeps UNAVAILABLE-SOURCE status, flows into SYNTHESIS.md advisory
- This mirrors the Phase 5 circuit breaker pattern — user decides, not automated retry

**12d: Update state.yml with Evaluator Results**
- Follow backup-before-write pattern: read current state.yml, cp state.yml state.yml.bak, modify in-memory, write entire file
- For each question, update: `status` (covered/partial/not_covered/unavailable_source), `gap_details` (null if covered, otherwise the evaluator's gap description)
- These are FINAL statuses replacing Phase 5's preliminary statuses

**12e: Display Assessment Summary**
- Show a summary table: question_id | priority | DA | status | coverage | corroboration | actionability
- Show counts: X COVERED, Y PARTIAL, Z NOT COVERED, W UNAVAILABLE-SOURCE
- If all COVERED: "All questions sufficiently covered. Proceeding to G2 gate."
- If gaps exist: "Gaps identified. G2 gate will determine next action."

<task type="auto">
  <name>Task 2: Append Step 13 — Dynamic Question Discovery</name>
  <files>skills/research/SKILL.md</files>
  <action>
Append Step 13 to the research SKILL.md immediately after Step 12.

Step 13 must implement the following:

**13a: Read Proposed Questions**
- Read `.expedite/research/proposed-questions.yml` (if it exists — may not exist if no agents proposed questions)
- If the file does not exist or is empty: display "No new questions discovered by research agents." and skip to Step 14
- Parse the `proposed_questions` list: each entry has `text`, `proposed_by`, `related_da`

**13b: Deduplicate via LLM Judgment**
- For each proposed question, compare semantically against:
  1. All existing questions in state.yml (current question plan)
  2. All other proposed questions (cross-deduplication)
- Use LLM judgment (not string matching) — questions can be semantically identical with different wording
- If semantically equivalent to an existing question: discard with note "Duplicate of {existing_question_id}"
- If semantically equivalent to another proposal: keep the one with better rationale/specificity
- This is LLM-based dedup performed inline in the main session (not a subagent)

**13c: Cap and Prioritize**
- After deduplication, take the top 4 unique proposals (RSCH-08 cap)
- Prioritize by: P0-related proposals first, then by relevance to gaps identified in Step 12
- If more than 4 unique proposals remain after dedup, keep only top 4

**13d: Present to User for Approval**
- Present each proposed question with its context: text, proposed_by (which batch/source), related_da
- Use a freeform prompt (not AskUserQuestion — these are too complex for yes/no): show the list and ask user to select which to approve (e.g., "Approve all / Approve 1,3 / Approve none / Modify")
- For approved questions: assign question_id (continuing from existing), set priority (suggest P1 unless user specifies), set status to "pending", set source to "discovered-round-{N}"
- Add approved questions to state.yml questions array (backup-before-write)
- Add approved questions to SCOPE.md question plan (append to appropriate DA section)

**13e: Display Discovery Summary**
- Show: "{X} questions proposed, {Y} duplicates removed, {Z} presented to user, {W} approved"
- If questions were approved: "Approved questions will be included in gap-fill research if G2 gate recycles."

Proceed to Step 14.
