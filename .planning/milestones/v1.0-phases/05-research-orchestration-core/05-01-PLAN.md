**Step 4: Form Source-Affinity Batches**
- Group questions by their primary source affinity:
  - For each question, look at source_hints from state.yml
  - If source_hints contains exactly one source AND that source is enabled in sources.yml, assign to that source's batch
  - If source_hints contains multiple sources, assign to the FIRST enabled source in the hints list (user can reassign in review)
  - If source_hints is empty or contains only disabled sources, assign to "web" as fallback (web is always available)
  - If a question status is already "covered" (from prior gap-fill round), SKIP it
- Create batch objects. Each batch has: batch_id (e.g., "batch-01"), source (e.g., "web"), source_tools (from sources.yml), and questions list
- Target: one batch per enabled source type. If only 1 question maps to a source, still give it its own batch (user can merge in review). Maximum 5 batches.
- For each question in the batch, include: id, text, priority, decision_area, evidence_requirements, source_hints

**Step 5: Validate DA Coverage**
- Extract all unique decision_area values from the questions in state.yml
- For each DA, verify at least one question is assigned to a batch (not skipped)
- If any DA has ZERO questions in batches, display WARNING: "Decision Area '{DA}' has no questions assigned to research batches. This DA will have no evidence for design decisions."
- Offer user the option to: (1) continue anyway, (2) cancel and modify scope first
- This check satisfies RSCH-14: every DA from scope has at least one research question covering it

**Step 6: Present Batch Plan for Approval**
- Display the batch plan in this exact format:
```
--- Research Batch Plan ---

Batch 1 ({source_name}) - {N} questions:
  [P0] {id}: {text} [DA: {decision_area}]
  [P1] {id}: {text} [DA: {decision_area}]
  ...

Batch 2 ({source_name}) - {N} questions:
  ...

--- {total_questions} questions across {total_DAs} DAs, {total_batches} batches ---
```

- After displaying, present the user with a freeform prompt:
  "Review the batch plan above. You can:
  - **approve** -- dispatch research agents with this plan
  - **modify** -- describe changes (e.g., 'move q03 to web batch', 'merge batches 2 and 3')
  - **cancel** -- abort research and return to scope"

- If user says "modify": apply the requested changes to the batch plan, re-display, and re-prompt. Loop until approved or cancelled.
- If user says "cancel": transition phase back to "scope_complete" (backup-before-write), decrement research_rounds, display "Research cancelled. Run `/expedite:scope` to modify your question plan." then STOP.
- If user says "approve": display "Batch plan approved. Proceeding to source validation..." and continue to Step 7 (next plan).

- **Evidence requirements flow-through**: When constructing the batch data structure for each question, ALWAYS include the full evidence_requirements string. This is critical -- research agents must receive typed evidence targets, not just question text. This satisfies RSCH-15.
