Create `hooks/lib/state-schema.js` that exports validation functions for all 5 state files. Each validator takes a parsed YAML object and returns `{ valid: boolean, errors: string[] }`.

**state.yml schema** (from PRODUCT-DESIGN.md):
Required fields: version (number), lifecycle_id (string|null), project_name (string|null), phase (string -- must be valid FSM phase), started_at (string|null), phase_started_at (string|null)
Valid phases: "scope_in_progress", "scope_complete", "research_in_progress", "research_complete", "design_in_progress", "design_complete", "plan_in_progress", "plan_complete", "execute_in_progress", "complete", "archived"
Note: The new state.yml does NOT contain questions, gate_history, tasks, current_task, current_wave, current_step, research_round, intent, description, imported_from, or constraints. Those move to their respective split files or are removed.

Wait -- re-reading the PRODUCT-DESIGN.md more carefully: state.yml contains version, lifecycle_id, project_name, phase, started_at, phase_started_at. That is 6 fields. But the current state.yml also has intent and description which are needed by skills. Looking at the consumption matrix in the design doc, state.yml is "always loaded" by all skills, so it should contain the identity fields that every skill needs.

Include these fields in state.yml validation:
- version (number, required)
- lifecycle_id (string, nullable for initialization)
- project_name (string, nullable for initialization)
- intent (string "product"|"engineering", nullable)
- description (string, nullable)
- phase (string, required, must be valid FSM value)
- started_at (string, nullable)
- phase_started_at (string, nullable)

**checkpoint.yml schema:**
Required fields when populated: skill (string), step (number|"complete"), label (string)
Optional fields: substep (string|null), continuation_notes (string|null), inputs_hash (string|null), updated_at (string|null)
Validation: step must be positive integer or the string "complete"

**questions.yml schema:**
Root key: `questions` (array)
Each entry: id (string, required), text (string, required), priority (string P0|P1|P2, required), decision_area (string, required), source_hints (array of strings), evidence_requirements (string, required), status (string: pending|covered|partial|not_covered|unavailable_source, required), source (string, required), gap_details (string|null), evidence_files (array of strings)

**gates.yml schema:**
Root key: `history` (array)
Each entry: gate (string G1-G5, required), timestamp (string, required), outcome (string: go|no-go|go-with-advisory|go_advisory|overridden, required), evaluator (string|null)
If outcome is "overridden": override_reason (string, required)
Optional fields: must_passed, must_failed, should_passed, should_failed, notes, overridden (boolean), structural_checks, semantic_evaluation

**tasks.yml schema:**
Root key: `tasks` (array), current_wave (number|null), current_task (string|null)
Each entry: id (string, required), title (string, required), status (string: pending|in_progress|complete|blocked|skipped|failed|partial, required), wave (number|null), assigned_agent (string|null)

Export: validateStateYml, validateCheckpointYml, validateQuestionsYml, validateGatesYml, validateTasksYml, and a `schemas` object containing the field definitions for reference.

Implementation notes:
- Use plain JavaScript (CommonJS with require())
- No TypeScript, no build step
- Each validator function: parse input -> check required fields -> check enum values -> return {valid, errors}
- Keep validators simple and declarative (~20-40 lines each per the design doc)
- Do NOT import js-yaml here -- validators receive already-parsed objects
