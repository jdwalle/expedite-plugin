**gates/g3-design.js** -- G3 design structural gate (Layer 1 of dual-layer):

Usage: `node gates/g3-design.js <project-dir>` where `<project-dir>` is the directory containing `.expedite/`.

Reads:
- `<project-dir>/.expedite/design/DESIGN.md`
- `<project-dir>/.expedite/scope/SCOPE.md`
- `<project-dir>/.expedite/state.yml` (for intent)

Uses shared utilities from `gates/lib/gate-utils.js` (same `require('./lib/gate-utils')` pattern as other gate scripts).

MUST criteria (all must pass for structural go):
- M1: DESIGN.md exists at `.expedite/design/DESIGN.md` and is non-empty
- M2: Every DA from SCOPE.md has a corresponding section in DESIGN.md. Extract DA identifiers from SCOPE.md (headings matching "DA-N" patterns). For each, verify DESIGN.md contains a heading or section referencing that DA identifier or DA name.
- M3: Each DA section in DESIGN.md contains at least one evidence citation. Evidence citations are references like "F1", "F2", "Finding 1", or file references to evidence-*.md or SYNTHESIS.md. Scan each DA section for at least one such reference. If a DA section has zero evidence citations, fail with: "DA-{N} section has no evidence citations."
- M4: DESIGN.md has correct format for intent. Read intent from state.yml. If "product": check for PRD-style sections (Problem Statement, User Flows, or similar). If "engineering": check for RFC-style sections (Background, Design, or similar). Flexible regex matching. If intent cannot be determined, pass with advisory.
- M5: DESIGN.md word count per DA section exceeds minimum threshold (100 words per DA section). If any DA section is below threshold, fail with: "DA-{N} section has only {W} words (minimum 100)."

SHOULD criteria (advisory, not blocking):
- S1: Each DA section contains a "Trade-offs" or "Alternatives" subsection
- S2: Each DA section contains a "Confidence" level indicator (High/Medium/Low or a score)
- S3: DESIGN.md total word count exceeds 500 words
- S4: If intent is "product", HANDOFF.md exists at `.expedite/design/HANDOFF.md`. Auto-PASS for "engineering" intent.

On completion: Same pattern as G1/G2/G4:
1. Compute outcome via `computeOutcome` (use "recycle" as holdOutcome per design skill convention)
2. Build entry via `buildEntry("G3", "g3-structural-script", ...)`
3. Append to gates.yml via `appendGateResult`
4. Print result to stdout via `printResult`
5. Exit 0 on go/go_advisory, exit 1 on recycle

**IMPORTANT:** This script is Layer 1 only. It writes its own entry to gates.yml with evaluator "g3-structural-script". The skill will later append a separate entry for the semantic layer (from gate-verifier). The structural script does NOT know about or invoke the semantic layer.

Each check produces actionable detail messages following the established pattern: "DA-2 (API Design) has no evidence citations in DESIGN.md" not "M3 failed".
