**Step 7: Pre-Validate Sources**
- Create the output directory: `mkdir -p .expedite/research` using Bash
- For each unique source in the approved batch plan:
  - If source is "web": SKIP validation (web is always available via WebSearch/WebFetch)
  - If source is "codebase": SKIP validation (codebase tools are always available)
  - If source is an MCP source (any key other than "web" or "codebase" in sources.yml):
    - Attempt a lightweight probe: call the first tool listed in that source's `tools` array from sources.yml with minimal parameters
    - If the tool call returns successfully or returns a server error (data accessible but query issue): mark source as AVAILABLE
    - If the tool call fails with a platform/connection error: mark source as UNAVAILABLE
- If any MCP source is UNAVAILABLE, display clear messaging:
  ```
  Source pre-validation failed:
    [FAIL] mcp: {source_name} -- {error_description}

  Options:
  1. Fix connection -- Resolve the issue and retry validation
  2. Reroute questions -- Move {affected_question_ids} to web search instead
  3. Skip questions -- Mark {affected_question_ids} as UNAVAILABLE-SOURCE and continue

  Which would you like to do?
  ```
- If user chooses "fix": re-validate the source. Loop until available or user picks another option.
- If user chooses "reroute": move affected questions to the web batch, re-display updated batch plan for confirmation.
- If user chooses "skip": update those questions' status to "unavailable_source" in state.yml (backup-before-write), remove them from the batch plan, and continue.
- If ALL sources validate (or user resolved all failures): display "All sources validated. Preparing agents..." and proceed.

**Step 8: Assemble Prompt Templates**
- For each batch in the approved plan, assemble the prompt:
  1. Determine the template file:
     - "web" source -> Read `skills/research/references/prompt-web-researcher.md`
     - "codebase" source -> Read `skills/research/references/prompt-codebase-analyst.md`
     - Any MCP source -> Read `skills/research/references/prompt-mcp-researcher.md`
  2. Extract frontmatter metadata (subagent_type, model) from the template
  3. Construct the `{{questions_yaml_block}}` -- a YAML block containing, for each question in the batch:
     ```yaml
     - id: "{id}"
       text: "{text}"
       priority: "{priority}"
       decision_area: "{decision_area}"
       evidence_requirements: "{evidence_requirements}"
     ```
  4. Replace ALL template placeholders:
     - `{{project_name}}` -> from state.yml project_name
     - `{{intent}}` -> from state.yml intent
     - `{{research_round}}` -> from state.yml research_rounds
     - `{{output_dir}}` -> ".expedite/research"
     - `{{output_file}}` -> ".expedite/research/evidence-{batch_id}.md" (e.g., evidence-batch-01.md)
     - `{{batch_id}}` -> the batch identifier
     - `{{timestamp}}` -> current ISO timestamp
     - `{{questions_yaml_block}}` -> the constructed YAML block
     - `{{codebase_root}}` -> current working directory (for codebase-analyst only)
     - `{{mcp_sources}}` -> YAML block of the MCP source config from sources.yml (for mcp-researcher only)
  5. **CRITICAL: After replacement, verify no `{{` patterns remain in the assembled prompt.** If any placeholder was not replaced, display error: "Template assembly failed: unreplaced placeholder found in {template_file}. Aborting dispatch." then STOP.
- Store each assembled prompt with its batch metadata (batch_id, source, question_ids, subagent_type).

**Step 9: Dispatch Parallel Subagents**
- Display "Dispatching {N} research agents in parallel..."
- For each batch, issue a Task() call **simultaneously** (all in parallel, per user decision):
  ```
  Task(
    prompt: {assembled_prompt},
    description: "Research {source} batch ({batch_id}): {comma-separated question ids}",
    subagent_type: "general-purpose"
  )
  ```
- **Maximum 3 concurrent agents** (RSCH-02). If more than 3 batches exist, dispatch the first 3 and queue remaining batches. After any agent completes, dispatch the next queued batch.
- As each agent completes, display a progress notification:
  ```
  Batch {N} complete ({source}) -- Evidence for {question_ids} written to evidence-{batch_id}.md ({remaining} remaining)
  ```
- If an agent fails (Task() returns error or agent reports critical failure): surface the failure to user with options:
  ```
  Agent failure in Batch {N} ({source}):
    Error: {error_description}

  Options:
  1. Retry -- Dispatch this batch again
  2. Skip -- Mark questions {question_ids} as failed and continue
  ```
  - If user retries, re-dispatch the same Task(). Maximum 1 retry per batch (per original RSCH-09 spirit, adapted for user-controlled recovery).
  - If user skips, mark those questions as "not_covered" in state.yml (backup-before-write).
- Continue until all batches are complete or user has addressed all failures.
  </action>

**Step 10: Collect Results and Update State**

- For each completed batch:
  1. Read the agent's condensed return summary (not the full evidence file -- avoid context bloat)
  2. From the summary, extract:
     - Per-question status: Did the agent find evidence meeting the requirements? (infer from GAPS section)
     - PROPOSED_QUESTIONS: Any new questions the agent discovered
     - CONFIDENCE level
  3. Update state.yml for each question in the batch (backup-before-write for EACH update cycle):
     - Set `evidence_files` to include the evidence file path (e.g., `[".expedite/research/evidence-batch-01.md"]`)
     - Set `status` based on agent findings:
       - If agent found evidence meeting requirements -> "covered" (preliminary -- Phase 6 sufficiency evaluator makes final determination)
       - If agent found partial evidence -> "partial"
       - If agent found no relevant evidence -> "not_covered"
       - If source was unavailable -> "unavailable_source"
     - Note: These are PRELIMINARY statuses. Phase 6 sufficiency evaluator will make the final assessment.
  4. Collect all PROPOSED_QUESTIONS from all agents into a single queue (do NOT present to user yet -- per user decision)
