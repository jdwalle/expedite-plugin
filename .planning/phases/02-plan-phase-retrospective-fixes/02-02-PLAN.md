Replace the S3 check logic (lines 319-355 in gates/g4-plan.js) to use the already-parsed `phases` array instead of its own inline line scanning.

The current bug: `inPhaseSection` starts as `false`, so ALL bullets before the first phase heading are flagged as orphans. Additionally, any `## ` non-phase heading within phase content resets `inPhaseSection` to false, potentially missing sub-section content.

**New S3 logic:**

Replace the entire S3 block (from the `// S3: No orphan tasks` comment through the `shouldResults.push` for S3) with:

```javascript
  // S3: No orphan tasks (task bullets should be under phase headings)
  // Only scan the region between the first and last phase headings.
  // Bullets before the first phase heading (metadata, coverage tables,
  // tactical decision tables) are not orphan tasks.
  var s3Passed = true;
  var s3Detail = '';
  if (planContent) {
    // Count tasks found by extractPhases (already parsed at line 114)
    var tasksInPhases = 0;
    for (var s3p = 0; s3p < phases.length; s3p++) {
      tasksInPhases += phases[s3p].tasks.length;
    }
    // Count all task bullets in the entire document using countAllTasks
    // (which only counts bullets after the first phase heading)
    var totalTaskCount = countAllTasks(planContent);
    var orphanCount = totalTaskCount - tasksInPhases;
    s3Passed = orphanCount <= 0;
    s3Detail = s3Passed
      ? 'All task items are under phase headings (' + tasksInPhases + ' tasks across ' + phases.length + ' phases)'
      : orphanCount + ' task-like bullet(s) found between phase sections but not within any phase';
  } else {
    s3Detail = 'Cannot check for orphan tasks: PLAN.md does not exist';
  }
  shouldResults.push({
    criterion: 'S3',
    passed: s3Passed,
    detail: s3Detail,
  });
```

This works because:
- `countAllTasks` (lines 96-111) already only counts bullets when `inPhase` is true (after a phase heading), and it never resets `inPhase` on non-phase headings -- it only turns ON at phase headings. So it counts all bullets from the first phase heading to end of file.
- `extractPhases` properly scopes content per-phase (content between phase heading N and phase heading N+1).
- The difference `countAllTasks - sum(phases[i].tasks)` catches bullets between phase sections (in the gap between one phase ending and the next beginning).
- Bullets BEFORE the first phase heading are not counted by either function, so they are correctly ignored.

Important: Keep the check as a SHOULD check pushing to `shouldResults` (not mustResults). Keep it as criterion 'S3'.

**Change 1: Loosen M6 heading regex in gates/g1-scope.js (lines 130-183).**

The current heading regex (line 138):
```javascript
if (/^#{1,4}\s+.*(?:DA-\d+|Decision\s+Area)/i.test(daLines[d])) {
```

This requires "DA-N" or "Decision Area" in the heading text. If Claude writes headings like `### Authentication Strategy (Depth: Deep)` without a DA-N label, no DA sections are detected and M6 fails.

Replace the DA detection logic. Keep the existing regex as the primary pass, then add a fallback pass for headings that contain depth/readiness patterns (indicating they are DA sections even without the DA-N label). Replace lines 133-141 (from `var daLines` through the heading detection loop closing brace) with:

```javascript
    var daLines = scopeContent.split('\n');
    var daHeadings = [];
    // Primary pass: headings with explicit DA-N or "Decision Area" markers
    for (var d = 0; d < daLines.length; d++) {
      if (/^#{1,4}\s+.*(?:DA-\d+|Decision\s+Area)/i.test(daLines[d])) {
        daHeadings.push({ line: d, text: daLines[d] });
      }
    }
    // Fallback pass: if no DA headings found via primary regex, look for
    // headings whose section content contains both depth and readiness
    // (these are DA sections even without the DA-N label)
    if (daHeadings.length === 0) {
      var candidateHeadings = [];
      for (var d2 = 0; d2 < daLines.length; d2++) {
        if (/^#{1,4}\s+/.test(daLines[d2]) && !/^#{1,4}\s+.*success\s+criteria/i.test(daLines[d2]) && !/^#{1,4}\s+.*source\s+config/i.test(daLines[d2]) && !/^#{1,4}\s+.*metadata/i.test(daLines[d2]) && !/^#{1,4}\s+.*project\s+context/i.test(daLines[d2]) && !/^#{1,4}\s+.*risk/i.test(daLines[d2]) && !/^#{1,4}\s+.*concern/i.test(daLines[d2])) {
          candidateHeadings.push({ line: d2, text: daLines[d2] });
        }
      }
      // Check each candidate: is the section content DA-like?
      for (var ch = 0; ch < candidateHeadings.length; ch++) {
        var chStart = candidateHeadings[ch].line;
        var chEnd = (ch + 1 < candidateHeadings.length) ? candidateHeadings[ch + 1].line : daLines.length;
        var chSection = daLines.slice(chStart, chEnd).join('\n');
        if (/depth/i.test(chSection) && /readiness/i.test(chSection)) {
          daHeadings.push({ line: chStart, text: candidateHeadings[ch].text });
        }
      }
    }
```

This approach:
- Preserves the existing primary regex (no regression for well-formatted headings)
- Only activates the fallback when the primary regex finds nothing
- Excludes known non-DA sections (Success Criteria, Source Config, Metadata, etc.)
- Validates candidates by checking section content for depth AND readiness patterns
- Keep ES5-compatible style (var, not const/let)

The rest of the M6 logic (lines 143-183: checking depth/readiness within each detected section) remains UNCHANGED.
