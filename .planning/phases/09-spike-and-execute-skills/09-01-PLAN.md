<objective>
Create the spike skill as a new 7th skill in the Expedite plugin. The spike skill reads a specific phase from PLAN.md, resolves its tactical decisions (inline or via optional focused research), and writes ordered implementation steps to SPIKE.md with full contract chain traceability.

Spike operates per-phase: `/expedite:spike 1` spikes Phase 1 only, `/expedite:spike 2` spikes Phase 2 only. The whole point of uniform phase sizing was to make each phase small enough for single-pass investigation.

Purpose: Spike bridges plan and execute -- it resolves "needs-spike" tactical decisions before implementation, preventing context-switching during execution. It is optional (execute works without it) and lightweight (no gate, no revision cycle).

Output: `skills/spike/SKILL.md` (8-step orchestration script) and `skills/spike/references/prompt-spike-researcher.md` (focused research prompt for Task() dispatch).
</objective>

**SKILL.md Steps 1-5** (~120-150 lines):

Start with the YAML frontmatter exactly as specified in the research (name, description with trigger phrases, allowed-tools including Task, argument-hint: "<phase-number>").

Then the !cat state.yml injection line (established pattern).

Then the role preamble: "You are the Expedite spike orchestrator. Your job is to resolve tactical decisions identified during planning and produce detailed implementation steps with full traceability. Spike is lightweight and optional -- no quality gate, no revision cycle. The user reviews your output and decides whether to proceed to execution. **Step 1: Prerequisite Check**
Look at injected lifecycle state.
- Case A: Phase is "plan_complete" -- display "Starting spike..." Proceed.
- Case B: Phase is "execute_in_progress" -- display "Spiking during execution..." (valid: user may spike Phase 2 while executing Phase 1). Proceed.
- Case C: Anything else -- display error with current phase, suggest running preceding skills. STOP.

**Step 2: Parse Phase Argument**
Extract the phase number from user input. Support flexible matching:
- Bare number: "1", "2", "3"
- "wave N" or "Wave N" (engineering intent)
- "epic N" or "Epic N" (product intent)
- If no argument provided and only one phase exists in PLAN.md, auto-select it
- If no argument and multiple phases exist, display available phases and ask user to choose
- Store the extracted phase number and determine the slug: `wave-{N}` for engineering, `epic-{N}` for product (based on intent from state.yml)

**Step 3: Read Plan Artifacts**
Read the following files:
1. `.expedite/plan/PLAN.md` -- the implementation plan
2. `.expedite/state.yml` -- extract project_name, intent, lifecycle_id
3. `.expedite/design/DESIGN.md` -- design decisions for traceability
4. If `.expedite/plan/override-context.md` exists -- read it, note affected DAs
5. **CRITICAL (per-phase context):** If spiking Phase N where N > 1, check for prior phase artifacts:
   - Read `.expedite/plan/phases/{prior-slug}/SPIKE.md` if it exists -- understand what was resolved in the previous phase
   - Read `.expedite/plan/phases/{prior-slug}/PROGRESS.md` if it exists -- understand what was already implemented
   - This enables Phase 2's spike to build on Phase 1's completed work as context

Display artifact loading summary (project name, intent, which phase is being spiked, whether prior phase context was found).

Error handling: If PLAN.md cannot be read, display error and STOP.

**Step 4: Extract Phase Definition**
From PLAN.md, find the matching wave/epic heading by number (e.g., "## Wave 2:" or "## Epic 2:").
Extract:
- Phase heading and description
- Design decisions covered (DA IDs)
- Tactical Decisions table (all TDs with classification)
- All tasks/stories in the phase (IDs, titles, design decision references, acceptance criteria)

If the requested phase is not found in PLAN.md, display available phases and ask user to clarify. STOP if still ambiguous.

Classify tactical decisions:
- resolved: list them, note "No investigation needed"
- needs-spike: these are the focus of Step 5

Display summary: "Phase {N}: {description} -- {X} tactical decisions ({Y} resolved, {Z} needs-spike), {W} tasks"

**Step 5: Resolve Tactical Decisions**
For each tactical decision in the phase:

If **resolved**: Display: "TD-{N}: {description} -- Resolved (from design decision DA-X: {brief}). No investigation needed."

If **needs-spike**: Present the user with options via freeform prompt:
```
TD-{N}: {description}
Alternatives: {list from PLAN.md}
DA reference: DA-X: {brief}

How would you like to resolve this?
> resolve inline (use your judgment) | research (dispatch focused investigation)
```

If user chooses **resolve inline**: Generate a resolution based on available design context and present it. "Resolution: {recommended approach}. Rationale: {reasoning from design decisions and context}."

If user chooses **research**:
1. Read `skills/spike/references/prompt-spike-researcher.md` (use Glob with `**/prompt-spike-researcher.md` if direct path fails)
2. Fill placeholders: {{tactical_decision}}, {{alternatives}}, {{da_reference}}, {{output_path}} (output to `.expedite/plan/phases/{slug}/spike-research-td-{N}.md`)
3. Create the output directory: `mkdir -p .expedite/plan/phases/{slug}/`
4. Dispatch via Task() with the filled prompt
5. Read the returned summary and present to user
6. Record: "Resolution: {recommendation from research}. Evidence: {output_file_path}"

After all TDs resolved, display summary: "All tactical decisions resolved. Generating implementation steps..."
