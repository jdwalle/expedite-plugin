<!-- Thinning pattern established by Plan 01 (scope + research skills) -->

The executor should read the Plan 01 output (thinned scope and research skills) to understand the pattern:
1. Compressed step-tracking: single reference instruction at top, not repeated per step

<!-- Gates.yml append pattern -->

Each gate write must READ existing gates.yml first and APPEND to the history array.
Multiple gate entries accumulate across the lifecycle:
- G1 (scope) -> G2 (research) -> G3 (design) -> G4 (plan) -> G5 (spike)
All entries share the same schema:
```yaml
history:
  - gate: "{G3|G4|G5}"
    timestamp: "{ISO 8601 UTC}"
    outcome: "{go|go_advisory|recycle|override}"
    evaluator: "{design-skill|plan-skill|spike-skill}"
    must_passed: {N}
    must_failed: {N}
    should_passed: {N}
    should_failed: {N}
    notes: "{summary or null}"
    overridden: false
```

Plan skill dispatches plan-decomposer with:
- project_name, intent, design_file path, scope_file path
- Override context path (if exists)
- Output path: .expedite/plan/PLAN.md

Spike skill dispatches spike-researcher with:
- project_name, intent, tactical_decision details, alternatives, da_reference
- Output path: .expedite/plan/phases/{slug}/spike-research-td-{N}.md

Execute skill dispatches task-implementer with:
- task definition (from SPIKE.md or PLAN.md), file targets, acceptance criteria
- Design decision reference for verification
- No output path (agent writes directly to project source files)
